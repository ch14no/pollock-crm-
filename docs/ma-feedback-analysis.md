## 1. エグゼクティブサマリ

## 2. 重要な発見（前提の修正）

### 2-1. すでに実現済みのもの

| 項目 | 状況 |
|---|---|
| ⑥ ステータス細分化 | `008_seed_ma_pipeline_tabs.sql` で **売主タブ18ステージ＋失注／買主タブ15ステージ＋失注** がシード済み（コミットログ上「本番適用済み」）。内容はSlackスレッドの酒田さん提案と一致。さらに `/settings` からsuper_admin/managerがステージ・タブを自由に追加・改名・並び替え・削除できるUIも完成済み。**→ 残作業は本番での表示確認とM&A事業部への使い方案内のみ** |
| ㉑ 名刺OCR（前半） | 名刺の表裏写真→Claude Vision APIで8項目自動抽出は `/contacts/new` に実装済み。**→ 残りはEight/LinkedIn転記の扱いのみ（後述、API制約あり）** |
| ① の案件側 | 案件（商談）レベルの売主/買主分離はパイプラインタブとして実現済み。**未実装なのは「顧客台帳」レベルの売主/買主区分**（後述） |

### 2-2. ②の正体＝2つの構造的バグ（コードレベルで確認済み）

1. **カスタムフィールドの編集がDBに保存されない**: `/contacts/[id]` のカスタムフィールド編集はZustand（localStorage）を更新するだけで、DB保存関数 `upsertContactCustomValue` が呼ばれていない。別端末・別ユーザーには変更が見えない。
2. **保存失敗が無音で成功に見える**: `updateContact` はRLSに拒否されても0件更新で成功扱いになり（`.select()`なし）、さらに呼び出し側が `.catch(() => {})` でエラーを握りつぶす。編集権限判定（`isOwnDivision`）が「表示中の顧客の事業部」ではなく「現在アクティブな事業部タブ」で行われるため、タブがズレた状態では「編集できたように見えて保存されていない」が起きうる。

→ ユーザー体感の「登録すると編集できない」はこの2つで説明がつく可能性が高い。**どの機能追加より先に修正すべき**（データ信頼性の問題）。

### 2-3. 本番DBで要確認の事項（改修着手前に必須）

- `contacts.address` / `contacts.department` カラムの実在（マイグレーションに無く、手動ALTER前提。無ければ更新が常時失敗している可能性）
- `pipeline_tabs`（007〜009）の適用状態と、M&Aタブ・ステージ（008）の実データ
- 商品マスタが**localStorageのみでDB未保存**という事実（ユーザー間・端末間で共有されていない。金銭管理の前提としてテーブル化が必要）

## 3. 拡張性のための設計原則

1. **M&A固有のコードパスを作らない**。「事業部ごとに設定できる汎用機構＋M&A事業部向けの設定データ」で実現する（パイプラインタブ方式の踏襲）。
2. **カスタムフィールド機構をdealsへ展開する**。現状 `division_custom_fields`/`contact_custom_values` はcontacts専用。`entity_type`（contact/deal）列の追加＋`deal_custom_values` 新設で、⑦優先度の表示項目・⑯報酬欄・㉒希望条件などを「設定で増やせる項目」にできる。
3. **新機能はDBを唯一の真実源にする**。既存のZustand二重管理（商品マスタ・カスタム値のlocalStorage保存）は不具合の温床であり、新機能では踏襲しない。
4. **資料は「リンク中心＋ファイル添付は補助」**。M&Aの契約書・IM等は機密性が高く、アクセス制御実績のあるGoogle Drive運用を活かす。CRMは「案件⇔資料の紐づけ台帳」に徹し、ファイル実体のSupabase Storage保存は必要なものに限定する（容量コスト・漏洩リスクの最小化）。
5. **通知は記録（activities自動記録）→表示（通知欄）→配信（Slack）の3層で段階的に**。ステージ変更の自動記録はDBトリガーで行い、更新経路（カンバン/モーダル/将来のAPI）を問わず一元化する。
